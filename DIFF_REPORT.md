## 1. Ключевые изменения в mtproxymax.sh

### 1.1. Replication / HA (Master-Slave синхронизация) — ОСНОВНОЕ НОВОЕ

Это самый крупный блок изменений (~800 строк). Полностью новая подсистема репликации конфигурации:

- **Новые переменные:** `REPLICATION_ENABLED`, `REPLICATION_ROLE`, `REPLICATION_SYNC_INTERVAL`, `REPLICATION_SSH_PORT`, `REPLICATION_SSH_USER`, `REPLICATION_DELETE_EXTRA`, `REPLICATION_SSH_KEY_PATH`, `REPLICATION_EXCLUDE`, `REPLICATION_RESTART_ON_CHANGE`, `REPLICATION_LOG`
- **Новые файлы:** `replication.conf`, `.ssh/` директория, `mtproxymax-sync.sh`
- **Команды CLI:** `replication setup|status|add|remove|list|enable|disable|sync|test|logs|reset|promote`
- **Механизм:** systemd timer + rsync over SSH. Мастер пушит конфиг на slave-серверы с настраиваемым интервалом (по умолчанию 60 сек).
- **Защита:** `settings.conf` и `replication.conf` всегда исключены из синхронизации — роль slave не перезаписывается.
- **TUI меню:** новый пункт `[r] Replication` в главном меню.

### 1.2. Обновление TeleMT версии

- `3.3.32` (a383efc) -> `3.3.38` (e630ea0)

## 4. Рекомендации по переносу в ветку master

### Приоритет: ВЫСОКИЙ

1. **Replication подсистема** — полностью самодостаточный блок. Переносить целиком (секция 14b + CLI handlers + TUI меню). Не конфликтует с Native-логикой, так как работает на уровне управления конфигурацией, а не сборки engine.

2. **Флаг `/tmp/.mtproxymax_stopped`** — критичное улучшение стабильности. Перенести в:
   - `stop_proxy_container()` — установка флага + `docker update --restart=no`
   - `start_proxy_container()` — очистка флага
   - `auto_recover()` — проверка флага
   - Цикл health check — проверка флага перед алертом

3. **Очистка старых образов в self_update()** — полезно для обоих вариантов. Перенести блок удаления образов.

4. **Исправления grep (`--` перед паттерном)** — багфикс, перенести обязательно.

5. **escape_md() удаление экранирования скобок** — проверить, не влияет ли на Native-версию Telegram-бота.

### Приоритет: СРЕДНИЙ

6. **Обновление TeleMT до 3.3.38** — если Native-версия использует тот же upstream, обновить пиннинг. Если Native использует свой коммит — пропустить.

7. **Улучшение вывода ошибок docker run** — перенести для лучшей отладки.

8. **Предупреждение о conns < 5** — перенести в `secret_set_limits()`.

9. **Метрики в TUI (пункты 3 и 4)** — проверить, есть ли аналог в Native-версии. Если `show_metrics` существует — добавить live-режим.

10. **Подсказки ввода памяти и обновлённые рекомендации conns** — UX-улучшения, перенести.

### Приоритет: НИЗКИЙ

11. **Предупреждение при смене домена** — мелкое UX-улучшение.

12. **`|| true` в list_backups()** — перенести для консистентности.

### Примечание

Наша Native-ветка (`master`) содержит код для нативной сборки engine без Docker. Replication подсистема **не зависит от Docker напрямую** (кроме `docker restart` на slave), поэтому должна работать корректно. Однако следует проверить:

- Использует ли Native-версия те же пути (`/opt/mtproxymax/`)
- Совместим ли механизм `run_proxy_container` / `stop_proxy_container` в Native (может быть другая логика запуска)
- Не дублируется ли уже функционал репликации
