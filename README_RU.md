MTProxyMax — полнофункциональный менеджер MTProto прокси для Telegram, работающий на **движке telemt 3.x (Rust)**. Обёртывает движок интерактивным TUI, полноценным CLI, телеграм-ботом для удалённого управления, контролем доступа на уровне пользователей, мониторингом трафика, цепочками прокси и автоматическими обновлениями — всё в одном bash-скрипте.

<img src="main.png" width="600" alt="MTProxyMax Главное меню"/>

```bash
sudo bash -c "$(curl -fsSL https://raw.githubusercontent.com/SamNet-dev/MTProxyMax/main/install.sh)"
```

## Почему MTProxyMax?

Большинство инструментов для MTProxy дают тебе прокси и ссылку. И всё. MTProxyMax даёт **целую платформу управления**:

- 🔐 **Мультипользовательские секреты** с индивидуальными квотами трафика, лимитами устройств и датами истечения
- 🤖 **Телеграм-бот** с 17 командами — управляй всем с телефона
- 🗂️ **Репликация** — автоматическая синхронизация конфигурации на подчинённые серверы через rsync+SSH
- 🖥️ **Интерактивный TUI** — не нужно запоминать команды, управление через меню
- 📊 **Метрики Prometheus** — реальная статистика трафика на пользователя, не гадание через iptables
- 🔗 **Цепочки прокси** — маршрутизация через вышестоящие SOCKS5 для дополнительной приватности
- 🔄 **Автовосстановление** — определяет простои, перезапускает автоматически, оповещает в Telegram
- 🐳 **Готовые Docker-образы** — установка за секунды, а не минуты

## ✨ Возможности

### 🛡️ Обфускация FakeTLS V2

Трафик прокси выглядит идентично обычному HTTPS-трафику. Движок **Fake TLS V2** зеркально имитирует реальные сессии TLS 1.3 — профили на домен, реальные наборы шифров, динамические длины сертификатов и реалистичная фрагментация записей. SNI рукопожатия TLS указывает на маскирующий домен (например, `cloudflare.com`), делая его неотличимым от обычного веб-сёрфинга для любой системы DPI.

**Маскировка трафика** идёт дальше — когда не-Telegram клиент зондирует твой сервер, соединение перенаправляется на реальный маскирующий домен. Твой сервер отвечает точно так же, как cloudflare.com.

### 👥 Управление мультипользовательскими секретами

Каждый пользователь получает свой **секретный ключ** с человеко-читаемой меткой:

- **Добавление/удаление** пользователей мгновенно — конфиг перегенерируется и прокси перезагружается на горячую
- **Включение/отключение** доступа без удаления ключа
- **Ротация** секрета пользователя — новый ключ, та же метка, старая ссылка перестаёт работать
- **QR-коды** — сканируются прямо в Telegram

---

### 🔒 Контроль доступа на уровне пользователя

Детальные лимиты, применяемые на уровне движка:

| Лимит | Описание | Пример | Лучше для |
|-------|----------|---------|-----------|
| **Макс. подключений** | Одновременные TCP-соединения (~3 на устройство) | `15` | **Ограничение устройств** |
| **Макс. IP** | Уникальные разрешённые IP-адреса | `5` | Анти-шеринг / злоупотребление |
| **Квота данных** | Пожизненный лимит трафика | `10G`, `500M` | Честное использование |
| **Дата истечения** | Автоотключение после даты | `2026-12-31` | Временный доступ |

> **Совет:** Каждое приложение Telegram открывает **~3 TCP-подключения** (по одному на DC). Для ограничения устройств умножай на 3: `conns 15` ≈ макс. 5 устройств. Установка ниже 5, скорее всего, сломает даже одно устройство. Лимиты IP менее надёжны, потому что мобильные пользователи перемещаются между вышками (кратковременно показывая 2 IP для 1 устройства), а несколько устройств за одним WiFi имеют 1 IP. Используй `ips` как вторичную меру.
>
> **Трафик и квоты пожизненные (накопительные)**, не ежемесячные. Они не сбрасываются автоматически. Используй `mtproxymax secret reset-traffic <метка>` для ручного сброса счётчиков или ротируй секрет.

```bash
mtproxymax secret setlimits alice 100 5 10G 2026-12-31
```

### 📋 Рецепты управления пользователями

<details>
<summary><b>Ограничение устройств на пользователя (рекомендуется)</b></summary>

```bash
mtproxymax secret setlimit alice conns 5    # Одно устройство (~3 подкл. на устройство, с запасом)
mtproxymax secret setlimit family conns 15  # Семья — до 5 устройств
```

Каждое приложение Telegram открывает ~3 TCP-подключения. Установка `conns 5` позволяет одно устройство с запасом. Если кто-то поделится ссылкой, второе устройство упрётся в лимит.

</details>

<details>
<summary><b>Уровни ограничения устройств</b></summary>

| Сценарий | `conns` | `ips` (опционально) |
|----------|---------|-------------------|
| Один человек, одно устройство | `1` | `2` (разрешить роуминг) |
| Один человек, несколько устройств | `3` | `5` |
| Малая семья | `5` | `10` |
| Малая группа / офис | `30` | `50` |
| Публичная/открытая ссылка | `0` | `0` (без ограничений) |

> Устанавливай `ips` немного выше `conns` для мобильного роуминга (переключение вышек временно показывает 2 IP для 1 устройства).

</details>

<details>
<summary><b>Ссылка для расшаривания с ограничением по времени</b></summary>

```bash
mtproxymax secret add shared-link
mtproxymax secret setlimits shared-link 50 30 10G 2026-06-01
```

Когда наступит дата истечения, ссылка перестанет работать автоматически.

</details>

<details>
<summary><b>Персональные ключи (рекомендуется)</b></summary>

```bash
mtproxymax secret add alice
mtproxymax secret add bob
mtproxymax secret add charlie

# Каждый человек получает свою ссылку — отключаем индивидуально
mtproxymax secret setlimit alice conns 10   # ~3 устройства
mtproxymax secret setlimit bob conns 5     # 1 устройство
mtproxymax secret setlimit charlie conns 15 # ~5 устройств
```

</details>

<details>
<summary><b>Отключение, ротация, удаление</b></summary>

```bash
mtproxymax secret disable bob    # Временно отключить
mtproxymax secret enable bob     # Восстановить доступ

mtproxymax secret rotate alice   # Новый ключ, старая ссылка умирает мгновенно

mtproxymax secret remove bob     # Навсегда удалить
```

</details>

### 🤖 Телеграм-бот (17 команд)

Полное управление прокси с телефона. Настройка занимает 60 секунд:

```bash
mtproxymax telegram setup
```

| Команда | Описание |
|---------|----------|
| `/mp_status` | Статус прокси, аптайм, подключения |
| `/mp_secrets` | Список всех пользователей с активными подключениями |
| `/mp_link` | Получить детали прокси + изображение QR-кода |
| `/mp_add <метка>` | Добавить нового пользователя |
| `/mp_remove <метка>` | Удалить пользователя |
| `/mp_rotate <метка>` | Сгенерировать новый ключ для пользователя |
| `/mp_enable <метка>` | Повторно включить отключённого пользователя |
| `/mp_disable <метка>` | Временно отключить пользователя |
| `/mp_limits` | Показать все лимиты пользователей |
| `/mp_setlimit` | Установить лимит пользователя |
| `/mp_traffic` | Разбивка трафика по пользователям |
| `/mp_upstreams` | Список цепочек прокси |
| `/mp_health` | Запустить диагностику |
| `/mp_restart` | Перезапустить прокси |
| `/mp_update` | Проверить обновления |
| `/mp_help` | Показать все команды |

**Автоматические оповещения:**
- 🔴 Прокси не работает → мгновенное уведомление + попытка авто-перезапуска
- 🟢 Прокси запущен → отправляет детали подключения + QR-коды
- 📊 Периодические отчёты по трафику с выбранным интервалом

### 🗂️ Репликация (синхронизация конфигурации мастер-подчинённый)

Автоматическая синхронизация нескольких серверов прокси. Мастер пушит изменения конфигурации на все подчинённые серверы через rsync+SSH с настраиваемым интервалом. Подчинённые получают `secrets.conf`, `upstreams.conf`, `instances.conf` и `config.toml` — их собственные настройки роли и локальное состояние никогда не перезаписываются.

**Настройка занимает две команды:**

```bash
# На мастере — запусти мастер, выбери Master, добавь подчинённый
mtproxymax replication setup

# На подчинённом — запусти мастер, выбери Slave
mtproxymax replication setup
```

**Как работает:**
- Мастер генерирует самодостаточный скрипт синхронизации в `/opt/mtproxymax/mtproxymax-sync.sh`
- Таймер systemd срабатывает каждые N секунд (по умолчанию: 60) и запускает синхронизацию
- При изменении — контейнер прокси на подчинённом автоматически перезапускается
- `settings.conf` и `replication.conf` всегда исключены — роль подчинённого никогда не перезаписывается

```bash
mtproxymax replication status     # Показать роль, состояние таймера, последнюю синхронизацию
mtproxymax replication sync       # Запустить немедленную синхронизацию
mtproxymax replication logs       # Просмотр журнала синхронизации
mtproxymax replication test       # Тест SSH-подключения ко всем подчинённым
mtproxymax replication promote    # Повысить подчинённый до мастера (аварийное переключение)
```

**Роли:**

| Роль | Описание |
|------|----------|
| **Master** | Пушит конфигурацию на подчинённые по расписанию |
| **Slave** | Получает конфигурацию, только чтение. Изменения нужно делать на мастере |
| **Standalone** | Репликация отключена (по умолчанию) |

### 🔗 Цепочки прокси (маршрутизация через вышестоящие)

Маршрутизация трафика через промежуточные серверы:

```bash
# Маршрутизировать 20% через Cloudflare WARP
mtproxymax upstream add warp socks5 127.0.0.1:40000 - - 20

# Маршрутизировать через резервный VPS
mtproxymax upstream add backup socks5 203.0.113.50:1080 user pass 80

# Поддерживаются хостнеймы (разрешаются движком)
mtproxymax upstream add remote socks5 my-proxy.example.com:1080 user pass 50
```

Поддерживает **SOCKS5** (с аутентификацией), **SOCKS4** и **прямую** маршрутизацию с балансировкой нагрузки на основе весов. Адреса могут быть IP или хостнеймами.

### 📊 Мониторинг трафика в реальном времени

Метрики Prometheus дают реальную статистику на пользователя:

```bash
mtproxymax traffic       # Разбивка по пользователям
mtproxymax status        # Обзор с количеством подключений
```

- Байты загруженные/скачанные на пользователя
- Активные подключения на пользователя
- Накопительное отслеживание между перезапусками

### 🌍 Геоблокировка

```bash
mtproxymax geoblock add ir    # Блокировать Иран
mtproxymax geoblock add cn    # Блокировать Китай
mtproxymax geoblock list      # Посмотреть заблокированные страны
```

Блоклисты CIDR на уровне IP применяются через iptables — трафик отбрасывается до достижения прокси.

### 💰 Монетизация через Ad-Tag

```bash
mtproxymax adtag set <hex_from_MTProxyBot>
```

Получи свой ad-tag у [@MTProxyBot](https://t.me/MTProxyBot). Пользователи видят закреплённый канал — ты зарабатываешь с прокси.

### ⚙️ Управление движком

```bash
mtproxymax engine status              # Текущая версия движка
mtproxymax engine rebuild             # Принудительная пересборка образа движка
mtproxymax rebuild                    # Принудительная пересборка из исходников
```

Обновления движка доставляются через `mtproxymax update`. Готовые мульти-архитектурные Docker-образы (amd64 + arm64) подтягиваются автоматически. Компиляция из исходников — автоматический резервный вариант.

<details>
<summary><b>Почему не mtg?</b></summary>

[mtg](https://github.com/9seconds/mtg) надёжен и минимален — по замыслу. Он **"сильно предвзят"** и намеренно базовый. Подходит для одноразового прокси по принципу «установил и забыл».

Но mtg v2 удалил поддержку ad-tag, поддерживает только один секрет, не имеет лимитов пользователей, интерфейса управления и автовосстановления.

</details>

<details>
<summary><b>Почему не официальный MTProxy?</b></summary>

[Официальный MTProxy от Telegram](https://github.com/TelegramMessenger/MTProxy) (реализация на C) **в последний раз обновлялся в 2019 году**. Без FakeTLS, без маскировки трафика, без контроля на уровне пользователя, ручная компиляция, без Docker.

</details>

<details>
<summary><b>Почему не простой Bash-установщик?</b></summary>

Скрипты типа MTProtoProxyInstaller устанавливают прокси и дают ссылку. И всё. Без управления пользователями, без мониторинга, без бота, без обновлений, без восстановления.

MTProxyMax — это не просто установщик — это **платформа управления**, которая заодно устанавливает себя.

</details>

## 🏗️ Архитектура

```
Telegram-клиент
      │
      ▼
┌─────────────────────────┐
│  Твой сервер (порт 443) │
│  ┌───────────────────┐  │
│  │  Docker-контейнер  │  │
│  │  ┌─────────────┐  │  │
│  │  │   telemt     │  │  │  ← Движок Rust/Tokio
│  │  │  (FakeTLS)   │  │  │
│  │  └──────┬──────┘  │  │
│  └─────────┼─────────┘  │
│            │             │
│     ┌──────┴──────┐     │
│     ▼             ▼     │
│  Прямая      SOCKS5     │  ← Маршрутизация через
│  маршр-я    цепочки     │     вышестоящие серверы
└─────────┬───────────────┘
          │
          ▼
   Серверы Telegram


Репликация мастер-подчинённый (опционально):

  Мастер-сервер             Подчинённые серверы
  ┌──────────────┐           ┌──────────────┐
  │ mtproxymax   │──rsync──▶ │ mtproxymax   │
  │ (systemd     │   +SSH    │ (получает    │
  │  таймер 60с) │           │  конфиг)     │
  └──────────────┘           └──────────────┘
```

| Компонент | Роль |
|-----------|------|
| **mtproxymax.sh** | Единый bash-скрипт: CLI, TUI, менеджер конфигурации |
| **telemt** | Движок MTProto на Rust, запущенный в Docker |
| **Служба телеграм-бота** | Независимая служба systemd, опрашивающая Bot API |
| **Служба синхронизации репликации** | Таймер systemd, пушащий конфигурацию на подчинённые серверы |
| **Конечная точка Prometheus** | `/metrics` на порту 9090 (только localhost) |

## 📋 Журнал изменений

### v1.0.4 — Репликация, движок v3.3.38, политика SNI и панель метрик

- **Репликация** — синхронизация конфигурации с мастера на подчинённые серверы через rsync+SSH с таймером systemd
- **Автоисключение** — `settings.conf` и `replication.conf` никогда не синхронизируются на подчинённые, сохраняя их роль и локальное состояние
- **Мастер** — интерактивная настройка для master, slave и standalone ролей
- **Защита подчинённого** — установка роли slave останавливает любой локальный таймер синхронизации и очищает устаревший список узлов
- **Повышение** — `mtproxymax replication promote` для ручного аварийного переключения (slave → master)
- **Защита ролей** — `replication add/remove/sync` блокируются с явной ошибкой на подчинённых серверах
- **Автообновление скрипта синхронизации** — ручная `sync` всегда перегенерирует скрипт из текущего бинарника

**Обновление движка (v3.3.31 → v3.3.38):**

- **Исправления Apple/XNU** — Устранение проблем подключения для iOS/macOS клиентов Telegram
- **Переработка ME** — Гибридный цикл маршрутизации, снимки ArcSwap, параллельные проверки здоровья, уточнённый карантин, многоуровневое обратное давление
- **Управление conntrack** — Автоматическое управление таблицей conntrack при нагрузке с настраиваемыми водяными знаками
- **Новые методы ретрансляции** — Улучшенный средний/прямой ретранслятор с адаптивными буферами
- **Быстрая инициализация ME2DC** — Быстрый запуск, неудержимая инициализация (теперь включена по умолчанию)
- **Оптимизации памяти** — Сокращение пула буферов, сжатие вектора сессий, уменьшение памяти статистики
- **Настройка таймаута вышестоящих** — Таймаут `tg_connect` теперь применяется к TCP-подключениям DC, предотвращает зависание соединений
- **API активных IP** — Новая конечная точка `/v1/stats/users/active-ips`
- **Резервный конфиг** — Движок грациозно fallback'ит при неудаче загрузки основного конфига

**Новые возможности:**

- **Политика неизвестного SNI** — Настраиваемая `mask` (разрешать, по умолчанию) или `drop` (отклонять) для TLS-соединений с несовпадающим SNI. TUI: Безопасность и маршрутизация > Политика неизвестного SNI. CLI: `mtproxymax sni-policy [mask|drop]` ([#40](https://github.com/SamNet-dev/MTProxyMax/issues/40))
- **Панель метрик движка** — `mtproxymax metrics` показывает подключения, маршрутизацию вышестоящих, статистику на пользователя и состояние пула ME. `mtproxymax metrics live` для автообновления
- **Сброс счётчиков трафика** — `mtproxymax secret reset-traffic <метка|all>` для ручного сброса накопительного трафика на пользователя

**Исправления ошибок:**

- **Исправление broken pipe на Alpine** — FIFO подстановки процессов заменены here-строками для избежания гонки SIGPIPE ([#37](https://github.com/SamNet-dev/MTProxyMax/issues/37))
- **Исправление двойного ввода на Alpine** — Очистка оставшихся байт escape-последовательностей от мультибайтовых нажатий клавиш в меню TUI ([#38](https://github.com/SamNet-dev/MTProxyMax/issues/38))
- **Исправление отклонения SNI после обновления движка** — По умолчанию `unknown_sni_action` изменено на `drop` в telemt v3.3.31+, теперь явно устанавливается в `mask` для обратной совместимости ([#40](https://github.com/SamNet-dev/MTProxyMax/issues/40))

### v1.0.3 — Заметки, применение квот, мультипорты и прочее

- **Заметки к секретам** — Прикрепление описаний к секретам (`secret note <метка> <текст>`, пункт TUI [8])
- **Предупреждения об истечении** — Список секретов показывает `(3д ост.)` / `(истекло)`, оповещения в Telegram за 3 дня до истечения
- **Применение квот** — Автоотключение секретов при 100% квоты, предупреждение на 80%, работает даже без телеграм-бота
- **Автоотключение при достижении квоты** — `secret reenable <метка>` для восстановления с опциональным сбросом трафика
- **JSON-статус** — `mtproxymax status --json` для интеграции с Grafana/Zabbix/мониторингом
- **Журнал подключений** — Журнал активности на пользователя с авторотацией (`connlog` CLI, TUI Журнал и трафик > [2])
- **Резервное копирование и восстановление** — `mtproxymax backup`, `restore <файл>`, `backups` — включает весь конфиг, секреты и статистику
- **Мультипортовые экземпляры** — Запуск прокси на нескольких портах с общими секретами (`instance add/remove/list`)
- **Горячая перезагрузка секретов** — Добавление/удаление/ротация секретов без разрыва активных соединений
- **Геоблокировка белого списка** — Разрешить только конкретные страны, заблокировать всё остальное ([#29](https://github.com/SamNet-dev/MTProxyMax/issues/29))

### v1.0.2 — Постоянный трафик и производительность TUI

- **Постоянные счётчики трафика** — Статистика трафика (ВХОД / ИСХОД) теперь переживает перезапуск контейнера ([#13](https://github.com/SamNet-dev/MTProxyMax/issues/13))
- **Всегда включённое отслеживание трафика** — Накопительный трафик сохраняется на диск каждые 60с, даже без включённого телеграм-бота
- **Предостановочный сброс трафика** — Финальный снимок трафика сохраняется перед каждой остановкой/перезапуском, без потери данных при чистом завершении
- **Пакетная загрузка статистики в TUI** — Единый запрос метрик + единое чтение файла заменяет порождение процессов на пользователя
- **Атомарная запись файлов с блокировкой** — Файлы трафика используют `flock` для предотвращения гонок между демоном и CLI
- **Исправление маппинга направлений** — Согласованное `from_client`=вход, `to_client`=исход по всем функциям

### v1.0.1 — Пакетное управление секретами

- **Пакетное добавление** — `secret add-batch <м1> <м2> ...` добавляет несколько секретов за один перезапуск ([#12](https://github.com/SamNet-dev/MTProxyMax/issues/12))
- **Пакетное удаление** — `secret remove-batch <м1> <м2> ...` удаляет несколько секретов за один перезапуск
- **Флаг `--no-restart`** — `secret add/remove/add-batch/remove-batch --no-restart` для скриптов и автоматизации
- **Опции TUI** — Интерактивные пункты меню [6] и [7] для пакетных операций

### v1.0.0 — Движок v3.3.30

**Обновление движка (v3.3.28 → v3.3.30):**

- **Переработка загрузчика TLS** — Адаптивный каскад профилей (Chrome → Firefox → TLS 1.2 → legacy) с кэшированием на цель, автоматический fallback при рукопожатии — значительно сложнее отпечатать/заблокировать через DPI
- **Валидатор SNI TLS** — Обеспечивает SNI из настроенного домена, дёшево отбрасывает некорректные зонды
- **Атомарные квоты на пользователя** — Удалена блокировка из горячего пути для лучшей пропускной способности под нагрузкой
- **Доверенные CIDR протокола PROXY** — Ограничение доверия к заголовку PROXY определёнными сетями-источниками
- **Усиление уклонения от DPI** — Усиление формы/тайминга, маскировка префетча, защита от крошечных кадров
- **Вышестоящий Shadowsocks** — Новый вариант транспорта вышестоящего наряду с SOCKS5
- **Антизалипание ME + сторожевой таймер сирот** — Исправление первопричины случайных отключений
- **Адаптивные буферы** — Динамический размер буферов: меньше ОЗУ при малой нагрузке, больше пропускной способности при высокой
- **Производительность потока** — В 3 раза быстрее сброс D2C + немедленный сброс ACK для меньшей задержки
- **ME, управляемый событиями** — Пул переключается с активного опроса на управляемый событиями, снижая потребление CPU в простое
- **Перенаправление ME/DC** — Динамическое перенаправление при деградации предпочитаемого пути дата-центра
